Architecture: Image-to-World Diffusion

Core pipeline

  1. Encode the current frame as the visual state
  2. Encode keyboard and mouse actions as control tokens
  3. Condition a 17B diffusion backbone on state plus action
  4. Decode the next world frame with temporal consistency

Why diffusion?

  • Strong visual fidelity under complex texture variation
  • Better local coherence than autoregressive pixels
  • Natural fit for conditional control signals

Model contract

Input state + player intent -> next playable frame

Describe the model as an interactive conditional diffusion system rather than a generic video generator. The essential idea is state plus action into next frame, repeated over time. Mention 17B parameters only once here; the audience mainly needs the control loop. Transition to the action channel because that is what makes the system interactive.

Action Conditioning: Keyboard + Mouse

Fine-grained control

Matrix-Game uses the same primitives a player uses: movement keys plus mouse deltas. This makes controllability legible and benchmarkable.

Control Objective

Native player input

No abstract action space. The benchmark tests the same keyboard and mouse signals humans actually produce.

Stress that Matrix-Game is not controlled through latent knobs or synthetic command labels. It takes real keyboard and mouse input, which makes both the demos and the evaluation much more convincing. Training Data: Matrix-Game Dataset

Dataset pipeline

MineDojo gameplay traces are paired with fine-grained player actions, then converted into frame-action training tuples for supervised world modeling.

1. Collection

Minecraft trajectories capture varied scenes, motion, and interactions.

2. Annotation

Keyboard states and mouse deltas are aligned to frames.

3. Training Pairs

Current frame plus action predicts the next frame.

This slide explains why the model can be controllable at all: the dataset preserves action semantics at frame level. Call out the value of aligned mouse and keyboard signals, since that is the unusual part of the data engine. Transition into the demo slide to show the payoff.
