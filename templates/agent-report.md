# Agent Report Template

Use this template for every agent report in the review pipeline. Keep the structure stable so reports can be aggregated, diffed across rounds, and verified mechanically.

```markdown
## [Agent Name] Report: [Deck Name]

**Date:** YYYY-MM-DD
**Agent:** [agent type — slide-critic | layout-auditor | typography-reviewer | verifier | etc.]
**Round:** N (if applicable, omit for single-pass reviews)
**File:** [path to .qmd file]
**Verdict:** APPROVED | NEEDS REVISION | REJECTED

---

## Hard Gate Status

Non-negotiable quality conditions. A single FAIL = REJECTED verdict.

| Gate | Status | Detail |
|------|--------|--------|
| Compilation | PASS / FAIL | `quarto render` exit code |

| Image references | PASS / FAIL | Broken refs: [list or "none"] |
| YAML frontmatter | PASS / FAIL | Missing fields: [list or "none"] |
| Cross-references | PASS / FAIL | Broken refs: [list or "none"] |
```

## Agent-Specific Notes

### Review Agents
- Populate `Hard Gate Status`, `Score`, `Issues Found`, `Strengths`, and `Recommended Next Actions`
- `Strengths` must not be empty
- Report concrete slide references, not generic observations

### Fix Agents
- Keep finding IDs from the source reports unchanged when referencing prior findings
- Explain how each fix maps back to the incoming report
- If a finding could not be fixed, state that explicitly and explain why

### Verification Agents
- Treat any compilation failure as `Verdict: REJECTED`
- Record fix verification, not new design opinions
- Confirm whether each previously reported issue is resolved, unresolved, or regressed

## Conventions

- Use one finding ID per issue and keep it stable across rounds
- Use slide numbers, not vague references like "middle section"
- If a section has no items, say so explicitly
- Scores must add up exactly
- Keep recommendations actionable and bounded
